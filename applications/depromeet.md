## Q1. 디프만에 지원하게 된 동기와 이번 기수 활동을 통해 이루고 싶은 구체적인 목표 (500자)

산업기능요원으로 서울소프트에서 Django 기반 ERP 시스템을 풀스택으로 개발하며 실무 역량을 쌓고 있습니다. 동시에 개인적으로 React, TypeScript, Next.js를 활용해 4개월간 7개의 웹 서비스를 기획부터 배포까지 완성했습니다. 토스 앱인토스 미니앱 배포, Firebase 실시간 시스템 구축 등 빠르게 만들고 배포하는 경험은 충분히 쌓았지만, 대부분 1인 개발이었기에 체계적인 팀 협업 경험이 부족하다는 한계를 느꼈습니다. 해커톤에서 2인 팀으로 장려상을 수상하며 협업의 가치를 체감했고, 디자이너·개발자가 함께 서비스를 릴리즈하는 디프만이야말로 이 갈증을 해소할 수 있는 환경이라 확신하여 지원하게 되었습니다.
이번 기수를 통해 이루고 싶은 목표는 다양한 직군의 팀원들과 소통하며 사용자 관점의 서비스 설계 역량을 키우는 것입니다. 저는 웹 개발 전반의 실무 경험과 빠른 실행력을 팀에 기여하고, 팀원들로부터 새로운 시각과 자극을 받으며 함께 성장하고 싶습니다.

## Q2. 병행 활동 및 시간 관리 계획 (500자)

현재 병행 중인 활동은 (주)서울소프트 근무(산업기능요원)이며, 건국대학교 서울캠퍼스는 휴학 중입니다. 별도 스터디나 사이드 프로젝트 계획은 없어 디프만 활동에 온전히 집중할 수 있습니다.
회사는 평일 자율출근제로 운영되어 출퇴근 시간을 유연하게 조절할 수 있고, 필요 시 연차를 활용해 일정을 조율할 수 있습니다. 지난 기간동안 회사 업무와 개인 프로젝트 7개를 병행하며 배포한 경험이 있기에, 시간 관리에 자신이 있습니다. 당시에도 평일 업무 후 저녁과 주말 시간을 활용하는 루틴이 이미 체화되어 있었습니다.
구체적인 계획은 다음과 같습니다. 평일에는 업무를 집중 처리 및 필요 시 동아리 프로젝트에 참여하며, 주말은 정기 세션 참석과 팀 프로젝트에 할애하고자 합니다.
시간이 부족하다는 핑계 대신 새벽과 주말을 활용하는 것이 이미 습관이 된 만큼, 끝까지 책임감 있게 참여하겠습니다.

## Q3. 예상치 못한 어려움을 극복하고 완주한 경험 (500자)

코레일 해커톤에서 2인 팀으로 참가했을 때, 추천 정확도를 높이기 위해 사용자에게 많은 정보를 입력받는 백엔드 구조를 설계했습니다. 그런데 프론트엔드 개발자가 입력 항목이 많아질수록 사용자 경험이 크게 저하된다고 강하게 반대했습니다. 실제로 사진 촬영 전 단계에서 입력해야 할 정보가 과도해지며 서비스 흐름이 끊기는 문제가 발생했습니다.
각자 왜 그 방식이 좋다고 생각하는지 근거를 솔직하게 공유하고, 현실적인 해결책을 함께 모색했습니다. 단순히 UI에서 항목을 줄이는 것이 아니라 데이터 흐름 자체를 재설계하기로 했습니다. 더 다양한 정보를 가지고 있는 API를 도입해 추천 범위를 먼저 축소하고, 기존 공공 데이터 API와 결합하는 구조로 변경했습니다. 그 결과 사용자 입력 부담을 줄이면서도 추천 정확도와 응답 속도를 함께 개선할 수 있었습니다.
이 경험을 통해 의견이 다를 때 감추지 않고 근거와 함께 이야기하는 소통이 팀의 신뢰를 쌓고, 더 좋은 결과물로 이어진다는 것을 배웠습니다.

## Q4. 주어진 역할 범위를 넘어 주도적으로 해결한 경험 (700자)

서울소프트에서 운수사 ERP 시스템을 유지보수하던 중, 라이브 서버에서만 배차 저장과 기사 교체가 실패하는 문제가 발생했습니다. 이 모듈은 제가 담당하는 영역이 아니었지만, 고객사 업무가 마비되는 긴급 상황이었기에 직접 원인 파악에 나섰습니다.
로컬에서는 정상 동작했고 서버 로그에도 500 에러가 남지 않아 원인 추적이 어려웠습니다. 브라우저 네트워크 탭을 하나하나 확인한 끝에, 요청 자체가 애플리케이션 레이어에 도달하기 전 Nginx에서 414 URI Too Long 에러로 차단되고 있음을 발견했습니다. 원인은 HTMX가 방대한 기사 ID 배열을 GET 방식으로 전송하며 URL 길이가 한도를 초과한 것이었습니다.
해결을 위해 세 가지 작업을 진행했습니다. 첫째, 요청 방식을 GET에서 POST로 전환하고 Django 뷰를 수정했습니다. 둘째, JavaScript에서 Set을 활용한 중복 제거 로직을 추가해 전송 데이터 크기를 줄였습니다. 셋째, 쿼리스트링 리다이렉트 대신 HTMX의 HX-Redirect 헤더를 활용해 처리 방식을 개선했습니다.
이 수정 이후 해당 모듈의 장애가 완전히 해소되었습니다. 해당 프로젝트의 담당자가 아니었지만 고객사의 불편을 방치할 수 없었습니다. 이후 팀 내에서 데이터 규모에 따른 환경 차이를 점검하는 코드 리뷰 항목이 추가되는 계기가 되었습니다.

## Q5. 빠른 실행과 시도를 통해 더 나은 결과를 만든 경험 (700자)

두바이 쫀득쿠키가 유행하던 시기, '먹방 체험형 웹앱을 만들면 재밌겠다'는 아이디어가 떠올랐습니다. 트렌드는 빠르게 식기 때문에 완벽한 기획서를 작성하는 대신, 당일 바로 React와 TypeScript로 개발에 착수했습니다. 핵심 기능인 터치 인터랙션과 애니메이션만 먼저 구현하고, 이틀 만에 첫 버전을 배포했습니다.
배포 후 토스 앱인토스 심사에서 첫 리젝을 받았습니다. 콘텐츠 완성도와 사용자 경험이 부족하다는 피드백이었습니다. 이를 계기로 Firebase Realtime Database를 연동해 실시간 동시 접속자와 통계 시스템을 추가했고, 파티클 애니메이션과 동적 대사 시스템으로 인터랙션을 개선했습니다. 수정 후 재심사를 통과해 일주일 만에 정식 배포할 수 있었습니다.
이 경험이 발판이 되어 '빠르게 만들고, 피드백을 반영해 개선하는' 사이클을 반복했습니다. 이후 Smoke Trace, Kickoff 등 여러 서비스를 같은 방식으로 완성하며, 매번 이전 프로젝트의 시스템을 확장·응용하고 새로운 기술적 도전을 더했습니다. Firebase 실시간 시스템을 Smoke Trace에서 확장했고, FootHub의 API 연동 경험을 Kickoff에서 더 복잡한 구조로 발전시켰습니다.
이 과정에서 완벽한 계획은 존재하지 않는다는 것을 배웠습니다. 빠르게 실행하고 피드백으로 방향을 수정하는 것이 더 좋은 서비스를 만든다는 것을 체득했습니다.

## Q6. 웹 개발에서 중요하다고 생각하는 기술적 요소 (최대 1000자)

웹 개발에서 가장 중요하다고 생각하는 요소는 '상태 관리와 데이터 패칭의 설계'입니다. UI가 아무리 좋아도 데이터 흐름이 꼬이면 사용자 경험은 무너지기 때문입니다.
Kickoff 프로젝트에서 이를 가장 깊이 고민했습니다. 유럽 5대 리그, K리그, UEFA 대회 등 800개 이상의 리그 데이터를 외부 API에서 가져와 실시간 스코어, 순위표, 선수 통계, 대진표 등 다양한 화면에 뿌려야 했습니다. 단순히 각 컴포넌트에서 fetch를 호출하면 동일한 데이터를 중복 요청하게 되고, 페이지 전환 시마다 로딩이 반복되어 사용성이 크게 떨어졌습니다.
이를 해결하기 위해 TanStack React Query를 도입했습니다. staleTime과 cacheTime을 데이터 특성에 맞게 분리 설정했습니다. 실시간 경기 스코어는 30초마다 자동 리페칭하고, 리그 순위나 선수 프로필처럼 변동이 적은 데이터는 캐시를 길게 유지해 불필요한 API 호출을 줄였습니다. 또한 queryKey를 리그·날짜·팀 단위로 세분화하여 한 화면에서 데이터를 갱신해도 다른 화면의 캐시가 영향받지 않도록 설계했습니다.
두바이 왕자와 Smoke Trace에서는 다른 관점의 상태 관리를 경험했습니다. Firebase Realtime Database로 동시 접속자 수와 누적 통계를 실시간 동기화해야 했는데, 여러 사용자가 동시에 값을 갱신할 때 데이터 정합성 문제가 발생했습니다. Firebase Transaction을 적용해 원자적 업데이트를 보장했고, onDisconnect 핸들러로 브라우저 종료 시에도 접속자 수가 정확히 차감되도록 구현했습니다.
이 경험들을 통해 상태 관리는 단순히 라이브러리 선택이 아니라, 데이터의 성격과 갱신 주기, 공유 범위를 분석하고 적절한 전략을 설계하는 것이 핵심이라는 것을 배웠습니다. 서버 상태와 클라이언트 상태를 명확히 분리하고, 각각에 맞는 도구와 패턴을 적용하는 것이 안정적이고 반응성 높은 웹 서비스의 기반이라고 생각합니다.

## Q7. 시행착오 끝에 구현에 성공한 기능 (최대 1000자)

두바이 왕자 프로젝트에서 실시간 동시 접속자 수를 표시하는 기능을 구현하며 가장 많은 시행착오를 겪었습니다.
첫 번째 시도는 단순하게 접근했습니다. 사용자가 접속하면 Firebase에 카운트를 +1하고, 페이지를 떠나면 -1하는 방식이었습니다. 그러나 브라우저를 그냥 닫거나 네트워크가 끊기면 -1 이벤트가 발생하지 않아 접속자 수가 계속 누적되는 문제가 생겼습니다. 테스트에서는 동시 접속자가 50명으로 표시되는데 실제로는 아무도 없는 상황이 발생했습니다.
두 번째 시도에서 Firebase의 Presence 시스템을 도입했습니다. onDisconnect 핸들러를 등록하면 서버 측에서 연결 해제를 감지해 자동으로 정리해주는 구조입니다. 그런데 이번에는 동시 접속자 수가 음수로 떨어지는 새로운 문제가 나타났습니다. 여러 탭을 열었다가 동시에 닫으면 onDisconnect가 중복 실행되며 카운트가 과도하게 차감된 것이었습니다.
세 번째 시도에서 근본적으로 구조를 바꿨습니다. 숫자를 단순 증감하는 대신, 각 사용자 세션을 고유 키로 등록하고 연결이 끊기면 해당 키를 삭제하는 방식으로 전환했습니다. 동시 접속자 수는 등록된 키의 개수로 계산합니다. 여기에 Firebase Transaction을 적용해 여러 사용자가 동시에 통계를 갱신할 때도 데이터 정합성이 깨지지 않도록 원자적 업데이트를 보장했습니다.
이렇게 완성한 실시간 시스템은 이후 Smoke Trace에서 그대로 확장 적용할 수 있었습니다. 흡연 시뮬레이션의 동시 접속자와 누적 통계에 동일한 패턴을 재사용하되, 개인별 통계 추적 기능을 추가하는 방식으로 발전시켰습니다.
이 경험을 통해 얻은 문제 해결 방식은 '구조를 의심하라'는 것입니다. 첫 번째와 두 번째 시도는 기존 구조 위에서 버그를 패치하려 했지만, 근본적인 데이터 모델 자체를 바꿨을 때 문제가 깔끔하게 해결되었습니다. 코드를 고치기 전에 설계가 올바른지 먼저 점검하는 습관이 이때 생겼습니다.

## 면접 대비 - 예상 질문 & 답변

### "왜 서버 파트가 아니라 웹으로 지원했나요?"

> 회사에서는 Django 기반 백엔드를 주로 담당하고 있지만, 개인적으로 가장 열정을 쏟고 성장한 영역은 웹 프론트엔드입니다. 지난 4개월간 React, TypeScript, Next.js로 7개의 서비스를 직접 만들고 배포하면서, 사용자와 가장 가까운 곳에서 인터랙션을 설계하고 즉각적인 피드백을 받는 과정이 가장 즐거웠습니다.
>
> 디프만은 디자이너와 함께 서비스를 릴리즈하는 동아리입니다. 디자이너의 결과물을 실제 화면으로 구현하고, 사용자 경험을 직접 만들어내는 역할은 웹 파트가 가장 적합하다고 판단했습니다. 서버 파트로 들어가면 디자이너와의 접점이 줄어들고, 제가 디프만에서 얻고 싶은 "다양한 직군과의 협업 경험"이 반감될 수 있다고 생각했습니다.
>
> 오히려 백엔드를 이해하는 웹 개발자라는 점이 팀에 더 큰 기여가 된다고 생각합니다. API 설계를 할 때 프론트에서 쓰기 편한 구조를 제안할 수 있고, 서버 개발자와의 커뮤니케이션에서도 양쪽 맥락을 이해하니 브릿지 역할을 할 수 있습니다.

## 예상 기술 면접 질문

### React / 프레임워크

**Q. React Hook이란?**
> 함수형 컴포넌트에서 상태, 생명주기, 컨텍스트 등 React의 핵심 기능을 사용할 수 있게 해주는 **함수**입니다. React 16.8에서 도입되었고, 이전에는 클래스형 컴포넌트에서만 가능했던 것들을 함수형에서도 쓸 수 있게 되었습니다. `use`로 시작하는 함수(useState, useEffect, useMemo 등)가 Hook이며, 규칙이 2가지 있습니다: 1) 컴포넌트 **최상위**에서만 호출(조건문/반복문 안 X), 2) React 함수형 컴포넌트 또는 커스텀 Hook 안에서만 호출. 커스텀 Hook을 만들어 로직을 재사용할 수도 있습니다.

**Q. React의 Virtual DOM이란?**
> 실제 DOM을 직접 조작하면 느리기 때문에, React는 메모리에 가상의 DOM 트리를 만들어 놓고 상태가 바뀌면 새로운 Virtual DOM과 이전 Virtual DOM을 비교(diffing)해서 **변경된 부분만 실제 DOM에 반영**합니다. 이를 재조정(Reconciliation)이라 하며, DOM 조작을 최소화해 성능을 높이는 핵심 원리입니다.

**Q. 컴포넌트(Component)란?**
> UI를 독립적이고 재사용 가능한 조각으로 나눈 것입니다. React에서는 함수형 컴포넌트가 표준이며, props를 입력받아 JSX를 반환합니다. 예를 들어 Kickoff에서 `LeagueCard`, `MatchItem`, `StandingsTable` 등을 독립 컴포넌트로 분리해 여러 페이지에서 재사용했습니다.

**Q. props와 state의 차이는?**
> **props**는 부모 → 자식으로 전달되는 **읽기 전용** 데이터입니다. 자식이 직접 수정할 수 없습니다. **state**는 컴포넌트 **내부에서 관리**하는 변경 가능한 데이터로, state가 바뀌면 해당 컴포넌트가 리렌더링됩니다.

**Q. JSX란?**
> JavaScript 안에서 HTML처럼 UI를 작성할 수 있게 해주는 문법 확장입니다. 브라우저가 직접 이해하지 못하기 때문에 Babel/SWC 등의 트랜스파일러가 `React.createElement()` 호출로 변환합니다. `{}`로 JavaScript 표현식을 삽입할 수 있는 것이 핵심입니다.

**Q. useState란?**
> 함수형 컴포넌트에서 **상태(state)를 선언**하는 Hook입니다. `const [value, setValue] = useState(초기값)` 형태로 사용하며, `setValue()`를 호출하면 컴포넌트가 새 값으로 리렌더링됩니다. 두바이 왕자에서 파티클 색상, 모달 열림 상태 등을 useState로 관리했습니다.

**Q. useEffect란? 언제 쓰나?**
> 컴포넌트의 **사이드 이펙트**(데이터 페칭, DOM 조작, 이벤트 리스너, 타이머 등)를 처리하는 Hook입니다. 의존성 배열(`[]`)에 따라 실행 시점이 달라집니다: 빈 배열이면 마운트 시 1회, 값이 있으면 그 값이 변할 때마다 실행. **cleanup 함수**를 반환해 언마운트 시 정리(리스너 해제, 타이머 취소 등)하는 것이 중요합니다.

**Q. useMemo와 useCallback의 차이는?**
> 둘 다 **메모이제이션**(이전 결과를 기억)을 위한 Hook입니다.
> - **useMemo**: **값**을 메모이제이션. 의존성이 바뀌지 않으면 이전 계산 결과를 재사용. Kickoff에서 리그 순위 정렬 같은 무거운 연산에 사용.
> - **useCallback**: **함수**를 메모이제이션. 의존성이 바뀌지 않으면 같은 함수 참조를 유지. 자식 컴포넌트에 콜백을 넘길 때 불필요한 리렌더링 방지에 유용.

**Q. useRef란?**
> **리렌더링 없이** 값을 유지하거나, **DOM 요소에 직접 접근**할 때 사용하는 Hook입니다. `useRef(초기값)`으로 생성하고 `.current`로 접근합니다. useState와 달리 값이 바뀌어도 리렌더링이 발생하지 않습니다. 스크롤 위치 저장, input 포커스, 이전 값 기억 등에 활용합니다.

**Q. useContext란?**
> **props drilling**(여러 단계를 거쳐 props를 전달하는 것) 없이, 컴포넌트 트리 전체에서 데이터를 공유할 수 있게 해주는 Hook입니다. `createContext`로 Context를 만들고, `Provider`로 감싸면 하위 컴포넌트 어디서든 `useContext()`로 값을 꺼내 쓸 수 있습니다. 테마, 인증 정보 등 전역적인 데이터에 적합합니다.

**Q. React.memo란?**
> 컴포넌트를 감싸서 **props가 변경되지 않으면 리렌더링을 건너뛰게** 하는 고차 컴포넌트(HOC)입니다. 부모가 리렌더링돼도 자식의 props가 동일하면 이전 렌더 결과를 재사용합니다. Kickoff에서 리그 카드 리스트처럼 개수가 많은 컴포넌트에 적용해 성능을 개선했습니다.

**Q. 리렌더링이 발생하는 조건은?**
> 1) **state가 변경**될 때, 2) **props가 변경**될 때, 3) **부모 컴포넌트가 리렌더링**될 때, 4) **Context 값이 변경**될 때. 이 중 3번이 흔한 성능 이슈 원인으로, React.memo, useMemo, useCallback 등으로 불필요한 리렌더링을 방지합니다.

**Q. 생명주기(Lifecycle)란? 함수형 컴포넌트에서는?**
> 컴포넌트가 **마운트(생성) → 업데이트(상태/props 변경) → 언마운트(제거)**되는 과정입니다. 클래스형에서는 `componentDidMount`, `componentDidUpdate`, `componentWillUnmount` 메서드를 사용했지만, 함수형에서는 **useEffect 하나로** 모든 생명주기를 처리합니다.

**Q. key prop은 왜 필요한가?**
> React가 리스트를 렌더링할 때, 각 항목을 **고유하게 식별**하기 위해 필요합니다. key가 없거나 index를 key로 쓰면 항목 추가/삭제/정렬 시 React가 어떤 요소가 변경됐는지 정확히 판단하지 못해 **불필요한 DOM 조작**이 발생하거나 상태가 꼬일 수 있습니다. 고유한 id 값을 key로 사용하는 것이 좋습니다.

**Q. 조건부 렌더링 방법은?**
> 1) **삼항 연산자**: `{isLogin ? <Dashboard /> : <Login />}`, 2) **&& 연산자**: `{isAdmin && <AdminPanel />}` (truthy일 때만 렌더), 3) **early return**: 컴포넌트 함수 내에서 `if (!data) return <Loading />;`. 상황에 맞게 선택하며, 복잡한 조건은 early return이 가독성이 좋습니다.

**Q. React에서 상태 관리를 어떻게 했나요? 전역 상태는요?**
> 두바이 왕자, Smoke Trace 등 소규모 프로젝트에서는 useState와 props로 충분했습니다. Kickoff처럼 페이지가 많고 데이터가 복잡해지면서 TanStack Query로 서버 상태를 분리하고, 클라이언트 상태(UI 토글, 필터 등)만 useState/useContext로 관리했습니다. 서버 상태와 클라이언트 상태를 구분하는 것이 핵심이라고 생각합니다.

**Q. 재사용성을 높이기 위해 어떻게 하나요?**
> 크게 3가지 방법을 사용합니다. 첫째, **컴포넌트를 작은 단위로 분리**합니다. Kickoff에서 ScoreBoard, PlayerList, StatsChart를 독립 컴포넌트로 만들어 경기 상세, 팀 상세, 리그 페이지 등 여러 곳에서 재사용했습니다. 둘째, **props로 데이터를 주입**해서 특정 데이터에 종속되지 않는 범용 컴포넌트를 만듭니다. 같은 ScoreBoard 컴포넌트에 다른 경기 데이터만 넘기면 됩니다. 셋째, **커스텀 Hook으로 로직을 재사용**합니다. 여러 컴포넌트에서 같은 API 호출 로직이 필요할 때 Hook으로 분리하면 중복 코드를 줄일 수 있습니다. 포트폴리오에서도 `portfolio.ts`에 데이터만 추가하면 컴포넌트 수정 없이 카드와 모달이 자동 생성되는 데이터 기반 구조로 재사용성을 높였습니다.

**Q. useEffect를 어떻게 사용하고, 주의할 점은?**
> 컴포넌트 마운트 시 데이터 페칭, 이벤트 리스너 등록, 타이머 설정 등에 사용합니다. 주의할 점은 cleanup 함수를 반드시 반환해서 메모리 누수를 방지하는 것과, 의존성 배열을 정확히 설정하는 것입니다. 두바이 왕자에서 Firebase 리스너를 useEffect로 등록하고, cleanup에서 off()를 호출해 구독을 해제했습니다.

**Q. React의 렌더링 최적화를 위해 해본 것이 있나요?**
> Kickoff에서 800개 리그 데이터를 다루면서, 불필요한 리렌더링이 성능 이슈를 일으켰습니다. React.memo로 변경되지 않는 컴포넌트의 리렌더링을 방지하고, useMemo로 리그 순위 계산 같은 무거운 연산을 캐싱했습니다. 또한 스켈레톤 로딩 UI를 적용해 사용자 체감 속도를 개선했습니다.

Q: Next.js의 SSR, CSR, SSG의 차이점을 설명하고 어떤 상황에서 무엇을 사용할 것인가요?

답변 방향: CSR은 클라이언트에서 렌더링하여 빠른 인터랙션을 제공하고, SSR은 서버에서 렌더링하여 초기 로딩 속도와 SEO에 유리합니다.  SSG는 빌드 시점에 정적 페이지를 생성하여 가장 빠르며, ISR은 주기적으로 정적 페이지를 업데이트합니다. 프로젝트의 특성에 맞게 "검색 노출이 중요한 상업용 페이지는 SSR을, 관리자 대시보드처럼 보안과 빠른 반응이 중요한 곳은 CSR을 쓴다"는 논리를 준비하세요.

**Q. SSR, SSG, CSR의 차이를 설명해주세요.**
> 웹 페이지를 **어디서, 언제 생성하느냐**의 차이입니다.
> - **CSR (Client Side Rendering)**: 브라우저가 빈 HTML을 받고 **JavaScript로 화면을 그리는 방식**. Vite + React의 기본 동작이며, Kickoff 같은 실시간 앱에 적합합니다.
> - **SSR (Server Side Rendering)**: 사용자가 **요청할 때마다** 서버에서 완성된 HTML을 생성해 보내는 방식. Next.js에서 지원하며, 매 요청마다 최신 데이터가 필요한 페이지에 적합합니다.
> - **SSG (Static Site Generation)**: **빌드 시** 미리 HTML을 생성해두는 방식. Next.js에서 지원하며, 포트폴리오처럼 콘텐츠가 자주 바뀌지 않는 정적 사이트에 적합합니다.
>
> 정리하면 CSR은 프레임워크 없이도 가능한 기본 방식이고, SSR/SSG는 Next.js 같은 프레임워크가 제공하는 기능입니다.

**Q. 포트폴리오만 Next.js를 쓰고, 나머지는 Vite를 쓴 이유는?**
> 프로젝트 성격에 맞게 선택했습니다.
>
> **Kickoff, 두바이 왕자, Smoke Trace → Vite (CSR SPA)**
> 이 프로젝트들은 SSR이 필요 없는 이유가 명확합니다. 첫째, 데이터가 **실시간으로 계속 바뀝니다**. Kickoff는 축구 경기 스코어가 수십 초마다 갱신되고, 두바이 왕자는 동시접속자 수가 실시간 변동합니다. 서버에서 HTML을 미리 만들어봤자 즉시 낡은 데이터가 되기 때문에 의미가 없습니다. 둘째, **SEO가 필요 없습니다**. 축구 스코어 앱이나 파티클 애니메이션 사이트는 검색엔진에 노출될 필요가 없고, 로그인한 사용자가 직접 접속해서 쓰는 **앱 성격**입니다. 셋째, Firebase Realtime Database와 WebSocket 기반의 **클라이언트 중심 상호작용**이 핵심이라 브라우저에서 직접 렌더링하는 CSR이 자연스럽습니다. 여기에 Vite의 빠른 HMR 덕에 실시간 기능을 개발하면서 즉각적으로 확인할 수 있어 개발 생산성도 높았습니다.
>
> **포트폴리오 → Next.js (SSG)**
> 반면 포트폴리오는 콘텐츠가 거의 바뀌지 않는 **정적 사이트**입니다. SSG로 빌드 시 HTML을 미리 생성하면 **SEO 최적화**(구글 검색 노출)와 **초기 로딩 속도 향상**을 동시에 얻을 수 있습니다. 또한 Next.js의 파일 기반 라우팅, Image 최적화, Vercel 원클릭 배포도 포트폴리오에 딱 맞는 장점이었습니다.

**Q. React와 Next.js를 왜 선택했나요? Vue나 Angular는?**
> React는 생태계가 가장 크고, 컴포넌트 기반 설계에 자유도가 높습니다. Next.js는 파일 기반 라우팅, SSR/SSG 지원, Vercel 배포의 편의성 때문에 선택했습니다. Vue는 학습 곡선이 낮고 템플릿 문법이 직관적이지만, TypeScript와의 통합이나 대규모 프로젝트에서의 생태계는 React가 더 풍부하다고 판단했습니다.

### TypeScript

**Q. TypeScript를 왜 사용하나요? JavaScript와 비교하면?**
> 컴파일 타임에 타입 오류를 잡아줘서 런타임 에러를 줄일 수 있고, IDE 자동완성이 강력해져 개발 생산성이 올라갑니다. 특히 외부 API 응답 데이터의 타입을 interface로 정의해두면 잘못된 프로퍼티 접근을 미리 방지할 수 있습니다. Kickoff에서 API-Football의 복잡한 응답 구조를 타입으로 정의한 것이 큰 도움이 됐습니다.

**Q. interface와 type의 차이는?**
> interface는 객체 타입 정의에 특화되어 있고 extends로 확장, 선언 병합이 가능합니다. type은 유니온, 인터섹션 등 더 유연한 타입 조합이 가능합니다. 저는 객체 구조 정의에는 interface를, 유니온 타입이나 유틸리티 타입에는 type을 사용합니다.

### JavaScript 핵심 개념

**Q. 호이스팅(Hoisting)이란? var, let, const의 차이는?**
> 호이스팅은 변수/함수 **선언이 코드 실행 전에 스코프의 최상단으로 끌어올려지는** 현상입니다. `var`는 호이스팅 시 `undefined`로 초기화되어 선언 전에 접근해도 에러가 아닌 `undefined`가 나옵니다. `let`과 `const`도 호이스팅되지만 **TDZ(Temporal Dead Zone)**에 걸려서 선언 전에 접근하면 ReferenceError가 발생합니다. `var`는 함수 스코프, `let`/`const`는 블록 스코프(`{}`)이며, `const`는 재할당이 불가합니다. 현대 JavaScript에서는 `var`를 쓰지 않고 `const`를 기본, 재할당이 필요할 때만 `let`을 사용합니다.

**Q. 클로저(Closure)란?**
> 함수가 **자신이 선언된 환경(렉시컬 스코프)의 변수를 기억하고 접근**할 수 있는 현상입니다. 내부 함수가 외부 함수의 변수를 참조하면, 외부 함수가 실행을 마쳐도 그 변수가 메모리에 유지됩니다. React의 `useState`도 클로저를 활용합니다 — `setState`가 이전 state 값을 기억하고 있는 것이 클로저 덕분입니다. 주의할 점은 불필요한 클로저가 메모리 누수를 일으킬 수 있다는 것입니다.

**Q. 이벤트 루프(Event Loop)란?**
> JavaScript는 **싱글 스레드**(한 번에 하나의 작업만 처리)인데, 비동기 작업(API 호출, 타이머 등)을 처리하기 위한 메커니즘이 이벤트 루프입니다. 동작 순서: 1) **콜 스택**에서 동기 코드 실행 → 2) 비동기 작업은 Web API에 위임 → 3) 완료되면 **태스크 큐**에 콜백 등록 → 4) 콜 스택이 비면 이벤트 루프가 큐에서 콜백을 꺼내서 실행. `Promise`의 `.then()`은 **마이크로태스크 큐**에 들어가서 일반 태스크보다 우선 실행됩니다.

**Q. Promise와 async/await의 차이는?**
> 둘 다 비동기 처리 방법이며, async/await는 Promise의 **문법적 설탕(syntactic sugar)**입니다. Promise는 `.then().catch()` 체이닝으로 처리하고, async/await는 `const data = await fetch(url)` 처럼 동기 코드처럼 읽힙니다. Kickoff에서 API-Football 데이터를 가져올 때 async/await를 사용했는데, 여러 API를 순서대로 호출하거나 `Promise.all()`로 병렬 호출할 때 가독성이 훨씬 좋습니다.

**Q. 화살표 함수와 일반 함수의 차이는?**
> 가장 큰 차이는 **this 바인딩**입니다. 일반 함수는 호출 방식에 따라 this가 달라지지만, 화살표 함수는 **자신이 선언된 시점의 상위 스코프의 this**를 그대로 가져옵니다(렉시컬 this). React에서 이벤트 핸들러에 화살표 함수를 쓰는 이유가 이것 때문입니다 — this가 컴포넌트를 가리키도록 보장합니다.

**Q. 구조 분해 할당(Destructuring)과 스프레드 연산자(...)란?**
> 구조 분해는 객체/배열에서 값을 꺼내는 문법입니다. `const { name, age } = user;` 또는 `const [first, second] = arr;`. 스프레드는 배열/객체를 펼치는 문법으로, `const newArr = [...arr, newItem]`, `const newObj = { ...obj, name: "변경" }` 처럼 **불변성을 유지하면서** 새 값을 만들 때 사용합니다. React에서 state를 업데이트할 때 스프레드로 새 객체를 만드는 것이 필수 패턴입니다.

### TanStack Query / 데이터 패칭

**Q. TanStack Query의 staleTime과 cacheTime(gcTime)의 차이는?**
> **핵심 차이: staleTime은 "다시 fetch할지 말지", gcTime은 "캐시를 메모리에서 지울지 말지"를 결정합니다.**
>
> - **staleTime** (기본값 0): 데이터를 "신선하다(fresh)"고 간주하는 시간. fresh 상태면 컴포넌트가 마운트되거나 윈도우 포커스돼도 **API 재요청을 하지 않음**. staleTime이 지나면 데이터는 stale(낡은) 상태가 되고, 다음 트리거(마운트, 포커스, 네트워크 재연결 등) 시 백그라운드에서 refetch함. : 접근해도 타이머 리셋되지않음
> - **gcTime** (구 cacheTime, 기본값 5분): 해당 queryKey를 **아무 컴포넌트도 구독하지 않을 때** 캐시 데이터를 메모리에 유지하는 시간. 이 시간이 지나면 캐시가 가비지 컬렉션됨. 시간 내에 다시 구독하면 **캐시된 데이터를 즉시 보여주면서** 백그라운드에서 refetch. : 점근 시 타이머 갱신
>
> 쉽게 말하면, staleTime은 "이 데이터 아직 쓸만해? → 네, 그럼 API 안 부름", gcTime은 "아무도 안 쓰는 데이터를 얼마나 들고 있을까? → 5분 지나면 버림"입니다.
> Kickoff에서 실시간 스코어는 staleTime 30초(자주 바뀌니까), 리그 순위는 5분(잘 안 바뀌니까)으로 분리 설정했습니다.

**Q. TanStack Query 없이 데이터 패칭하면 어떤 문제가 있나요?**
> useEffect + fetch로 직접 구현하면 로딩/에러 상태 관리, 캐싱, 중복 요청 방지, 자동 리페칭, 포커스 시 갱신 등을 전부 직접 구현해야 합니다. Kickoff에서 20개 이상의 API 엔드포인트를 다루는데 이걸 일일이 관리하면 코드가 엄청나게 복잡해집니다.

### Firebase

**Q. Firebase Realtime Database와 Firestore의 차이는?**
> Realtime Database는 하나의 큰 JSON 트리 구조이고, 데이터 변경 시 실시간으로 클라이언트에 push합니다. Firestore는 컬렉션-문서 구조로 복잡한 쿼리가 가능합니다. 두바이 왕자에서는 단순한 카운터와 접속자 관리였기 때문에 Realtime Database가 적합했습니다. 복잡한 필터링이 필요했다면 Firestore를 선택했을 겁니다.

**Q. Firebase Transaction이 뭔가요? 왜 필요했나요?**
> Transaction은 read-modify-write를 원자적으로 수행하는 것입니다. 두바이 왕자에서 여러 사용자가 동시에 쿠키 섭취량을 갱신하면, 일반 write로는 마지막 쓰기가 이전 값을 덮어씁니다. Transaction을 쓰면 현재 값을 읽고 → 그 위에 계산하고 → 쓰는 과정이 원자적으로 보장돼서 데이터 정합성이 유지됩니다.

### HTTP / REST API

**Q. GET과 POST의 차이는? 언제 어떤 걸 쓰나요?**
> GET은 데이터 조회용으로 URL에 파라미터를 담고, 캐싱이 가능하고 멱등성이 보장됩니다. POST는 데이터 생성/수정용으로 body에 데이터를 담고, URL 길이 제한이 없습니다. 회사에서 HTMX가 기사 ID 배열을 GET으로 보내다가 URI Too Long 에러가 발생한 것도 GET의 URL 길이 제한 때문이었고, POST로 전환해서 해결했습니다.

**Q. CORS가 뭔가요? 왜 발생하고 어떻게 해결하나요?**
> 브라우저가 보안을 위해 다른 출처(도메인/포트/프로토콜)의 리소스 요청을 제한하는 정책입니다. 프론트(localhost:3000)에서 백엔드(localhost:8000)로 요청하면 출처가 달라서 CORS 에러가 발생합니다. 서버에서 Access-Control-Allow-Origin 헤더를 설정하거나, 개발 시에는 프록시를 설정해서 해결합니다.

**Q. HTTP 상태 코드 중 자주 쓰는 것은?**
> 200(성공), 201(생성 성공), 400(잘못된 요청), 401(인증 필요), 403(권한 없음), 404(없음), 414(URI 너무 김 - 회사에서 직접 겪음), 500(서버 에러). 특히 414는 보통 잘 안 나오는데, Nginx가 GET 요청의 URL 길이를 제한해서 발생하는 것을 실무에서 경험했습니다.

### 성능 / 최적화

**Q. 웹 성능 최적화를 위해 해본 것이 있나요?**
> 포트폴리오에서 프로젝트 이미지를 모두 PNG에서 WebP로 변환해 80~95% 용량을 줄였습니다. Kickoff에서는 TanStack Query 캐싱으로 중복 API 호출을 제거하고, 스켈레톤 UI로 체감 로딩 속도를 개선했습니다. 또한 IntersectionObserver로 뷰포트에 들어오는 섹션의 데이터를 프리페칭하는 방식도 적용했습니다.

**Q. 이미지 최적화는 어떻게 했나요?**
> PNG 스크린샷을 sharp-cli로 WebP로 변환했습니다. 충주 프로젝트 스크린샷 5장의 경우 총 3.5MB에서 408KB로 약 88% 감소했습니다. WebP는 같은 품질에서 PNG보다 25~35% 작고, 브라우저 지원도 충분합니다. Next.js의 Image 컴포넌트를 쓰면 자동 최적화도 가능합니다.

### CSS / Tailwind

**Q. Tailwind CSS를 왜 선택했나요? 단점은 없나요?**
> 클래스명을 짓지 않아도 되고, 디자인 시스템의 일관성을 유지하기 쉽습니다. 빠른 프로토타이핑에 유리해서 7개 프로젝트를 빠르게 만드는 데 큰 도움이 됐습니다. 단점은 클래스가 길어져서 가독성이 떨어질 수 있고, 복잡한 커스텀 애니메이션은 CSS를 직접 작성해야 합니다.

**Q. 반응형 디자인은 어떻게 구현했나요?**
> Tailwind의 반응형 프리픽스(sm, md, lg)를 활용했습니다. 모바일 퍼스트로 기본 스타일을 작성하고, md 이상에서 레이아웃을 변경하는 방식입니다. 포트폴리오는 모바일·태블릿·데스크톱 3단계 브레이크포인트를 설정했고, 프로젝트 카드는 모바일에서 세로 배치, 데스크톱에서 가로 배치로 전환됩니다.

### 브라우저 / 웹 기초

**Q. 브라우저에 URL을 입력하면 어떤 과정이 일어나나요?**
> 1) **DNS 조회**: 도메인(jyc-portfolio.vercel.app)을 IP 주소로 변환 → 2) **TCP 연결**: 서버와 3-way handshake → 3) **HTTP 요청**: 서버에 HTML 요청 → 4) **서버 응답**: HTML 파일 전송 → 5) **파싱**: HTML → DOM 트리, CSS → CSSOM 트리 생성 → 6) **렌더 트리** 생성 (DOM + CSSOM 합침) → 7) **레이아웃(Reflow)**: 각 요소의 위치/크기 계산 → 8) **페인트(Repaint)**: 화면에 그리기 → 9) **컴포지팅**: 레이어를 합쳐 최종 화면 표시. 이 과정을 **임계 렌더링 경로(Critical Rendering Path)**라고 합니다.

**Q. Reflow와 Repaint의 차이는?**
> **Reflow(레이아웃)**: 요소의 크기나 위치가 바뀔 때 레이아웃을 다시 계산하는 것. width, height, margin, padding 변경 시 발생. **비용이 큼**. **Repaint**: 레이아웃 변경 없이 색상, 배경 등 시각적 속성만 바뀔 때 다시 그리는 것. Reflow보다 비용이 적음. 애니메이션에서 `transform`과 `opacity`를 쓰는 이유는 이 속성들이 Reflow/Repaint 없이 **컴포지팅 단계에서만** 처리되어 성능이 좋기 때문입니다. Framer Motion이 내부적으로 `transform`을 활용합니다.

**Q. 웹 접근성(Accessibility)이란?**
> 장애를 가진 사용자도 웹 콘텐츠를 이용할 수 있도록 하는 것입니다. 시각 장애인은 **스크린 리더**로 웹을 탐색하는데, 이때 시맨틱 태그(`<nav>`, `<main>`, `<article>`)와 ARIA 속성(`aria-label`, `role`)이 중요합니다. 또한 키보드만으로 모든 기능을 이용할 수 있어야 하고, 색상 대비도 충분해야 합니다. 포트폴리오에서 시맨틱 태그로 섹션을 구분하고, 이미지에 alt 속성을 설정했습니다.

**Q. 코드 스플리팅(Code Splitting)이란?**
> JavaScript 번들을 **여러 청크(chunk)로 나눠서** 필요한 시점에 로드하는 기법입니다. 모든 코드를 하나의 파일로 번들링하면 초기 로딩이 느려지기 때문에, 라우트별 또는 컴포넌트별로 분리합니다. React에서는 `React.lazy()`와 `Suspense`로 구현하고, Next.js는 페이지 단위로 **자동 코드 스플리팅**을 해줍니다.

**Q. XSS와 CSRF란?**
> **XSS(Cross-Site Scripting)**: 공격자가 웹 페이지에 악성 스크립트를 삽입하는 공격. 사용자 입력을 그대로 HTML에 렌더링하면 발생. React는 JSX에서 자동으로 이스케이핑해줘서 기본적으로 방어됩니다. **CSRF(Cross-Site Request Forgery)**: 사용자가 로그인된 상태에서 악성 사이트가 대신 요청을 보내는 공격. 토큰 기반 인증이나 SameSite 쿠키 설정으로 방어합니다.

### Git / 협업

**Q. Git 브랜치 전략은 어떻게 사용하나요?**
> 개인 프로젝트에서는 main + feature 브랜치 정도로 단순하게 운영했고, 회사에서는 main/develop/feature 브랜치를 분리해서 사용합니다. feature 브랜치에서 작업 후 develop에 머지하고, 안정화된 후 main에 배포하는 흐름입니다.

**Q. 코드 리뷰 경험이 있나요?**
> 회사에서 PR 기반 코드 리뷰를 진행하고 있습니다. 414 에러 해결 이후에는 "대량 데이터 전송 시 GET/POST 방식 확인" 같은 체크리스트 항목이 추가됐습니다. 디프만에서 더 다양한 관점의 코드 리뷰를 경험하고 싶습니다.

### 프로젝트 심화

**Q. Kickoff에서 queryKey를 어떻게 설계했나요?**
> 계층적으로 설계했습니다. `['matches', leagueId, date]`처럼 리그와 날짜를 조합해서, 특정 리그의 특정 날짜 경기만 무효화하거나 갱신할 수 있도록 했습니다. 리그 순위는 `['standings', leagueId]`, 선수 정보는 `['player', playerId]`로 독립적으로 관리해서 한 화면 갱신이 다른 화면에 영향을 주지 않게 했습니다.

**Q. 두바이 왕자에서 파티클 애니메이션은 어떻게 구현했나요?**
> 클릭 좌표를 기준으로 10~15개 파티클을 생성하고, 각각에 랜덤 방향(각도), 속도, 크기, 수명을 부여합니다. requestAnimationFrame으로 매 프레임마다 위치를 업데이트하면서 중력(아래 가속)과 투명도 감소를 적용합니다. 수명이 끝나면 배열에서 제거해 메모리를 관리합니다. CSS 애니메이션 대신 JS를 쓴 이유는 랜덤 방향과 물리 시뮬레이션이 CSS로는 제어하기 어렵기 때문입니다.

**Q. 토스 앱인토스 배포 과정은 어떻게 되나요?**
> Vercel에 웹앱을 배포하고, 토스 앱인토스 개발자 포털에서 미니앱으로 등록합니다. 심사 과정에서 콘텐츠 완성도, 사용자 경험, 안정성 등을 검토합니다. 두바이 왕자는 첫 심사에서 리젝됐는데, Firebase 실시간 기능과 인터랙션을 보강한 후 재심사를 통과했습니다.

**Q. 포트폴리오 사이트의 데이터 기반 구조란?**
> `portfolio.ts` 파일 하나에 프로젝트, 경력, 학력, 스킬 등 모든 데이터를 정의하고, 컴포넌트에서는 이 데이터를 불러와서 렌더링만 합니다. 새 프로젝트를 추가할 때 컴포넌트를 수정할 필요 없이 데이터 파일에 객체 하나만 추가하면 자동으로 카드, 모달, 스크린샷이 생성됩니다.

### 디자이너 협업

**Q. 디자이너와 협업한 경험이 있나요?**
> 회사에서 디자이너와 함께 웹 서비스를 개발하고 있고, 해커톤(코레일 동행열차)에서도 디자이너와 협업했습니다. Figma에서 디자인을 전달받아 CSS로 구현하는 과정에서, 간격이나 폰트 크기 등 세부 스펙을 Figma의 Inspect 기능으로 확인하며 정확하게 맞췄습니다. 기술적으로 구현이 어려운 부분(복잡한 애니메이션, 성능 이슈)은 미리 공유하고 대안을 함께 논의했습니다.

**Q. Figma를 사용해본 경험이 있나요?**
> 네, 디자이너가 전달한 Figma 파일에서 컴포넌트 구조, 색상값, 간격, 폰트 스타일 등을 확인하며 개발합니다. Figma의 Auto Layout은 CSS의 Flexbox와 유사해서, 디자인 구조를 보면 컴포넌트 설계를 미리 파악할 수 있습니다. 개인 프로젝트에서도 개발 전에 Figma로 간단한 와이어프레임을 그려보고 시작하는 편입니다.

**Q. 디자이너가 요청한 것이 기술적으로 어려울 때 어떻게 하나요?**
> 먼저 **왜 어려운지 기술적 근거를 설명**하고, **대안을 함께 제시**합니다. 예를 들어 "이 애니메이션은 Reflow가 많이 발생해서 모바일에서 버벅일 수 있는데, transform 기반으로 비슷한 효과를 내면 성능을 유지할 수 있습니다" 같은 식입니다. 단순히 "안 됩니다"가 아니라 기술적 제약 안에서 최대한 디자인 의도를 살리는 방향을 찾는 게 중요하다고 생각합니다.

### CI/CD / 테스팅

**Q. CI/CD란 무엇인가요?**
> **CI(Continuous Integration)**: 코드 변경을 자주 메인 브랜치에 병합하고, 자동으로 빌드/테스트를 실행하는 것. **CD(Continuous Deployment)**: 테스트를 통과하면 자동으로 배포까지 하는 것. 현재 모든 프로젝트를 Vercel로 배포하는데, GitHub에 push하면 자동으로 빌드 → 배포되는 것이 CD입니다. 디프만에서 GitHub Actions를 활용한 CI/CD 파이프라인을 경험해보고 싶습니다.

**Q. 테스트 코드를 작성해본 경험이 있나요?**
> 아직 프론트엔드 테스트 코드를 체계적으로 작성해본 경험은 부족합니다. 하지만 그 필요성은 강하게 느끼고 있습니다. Kickoff에서 API 응답 구조가 바뀌었을 때 여러 컴포넌트가 동시에 깨진 적이 있는데, 타입 체크 외에 단위 테스트가 있었다면 더 빠르게 발견할 수 있었을 것입니다. 디프만에서 Jest, React Testing Library 등을 실전에 적용해보고 싶습니다.

### 행동 역량 / 디프만 심화

**Q. 디프만에 지원한 이유는?**
> 혼자 7개 프로젝트를 만들면서 기술은 늘었지만, **다양한 직군과의 협업 경험**이 부족하다는 것을 절실히 느꼈습니다. 디프만은 디자이너와 개발자가 함께 실제 서비스를 출시하는 경험을 할 수 있고, 코드 리뷰와 기술 공유 문화가 있어서 혼자서는 얻기 어려운 성장을 할 수 있다고 생각합니다. 특히 서로의 튜터와 튜티가 되는 문화가 매력적이었습니다.

**Q. 가장 몰입했던 경험은?**
> 두바이 왕자 프로젝트입니다. "한입만 쿠키 먹기" 해커톤에서 3일 만에 완성해야 했는데, Firebase Realtime Database로 실시간 동시접속자 기능을 구현하면서 Transaction, onDisconnect 등 처음 접하는 개념을 밤새 학습하고 적용했습니다. 결과적으로 토스 앱인토스에 출시까지 했고, 이 경험이 이후 Smoke Trace, Kickoff까지 이어지는 원동력이 됐습니다.

**Q. 팀원이 중도 하차(탈주)하면 어떻게 하나요?**
> 먼저 **남은 인원으로 현실적인 스코프 조정**을 합니다. 전체 기능을 유지하려고 무리하기보다, 핵심 기능에 집중해서 완성도를 높이는 것이 중요하다고 생각합니다. 회사에서 동료가 퇴사한 후 업무를 인수받은 경험이 있는데, 그때도 우선순위를 다시 정하고 핵심부터 처리하는 것이 효과적이었습니다. 중요한 건 **서비스를 끝까지 출시하는 것**입니다.

**Q. 디프만 활동과 회사 업무를 어떻게 병행할 건가요?**
> 현재 산업기능요원으로 자율출근제를 활용하고 있어 시간 조율이 가능합니다. 또한 연차와 반차를 전략적으로 활용할 수 있습니다. 혼자서 7개 프로젝트를 퇴근 후와 주말에 만든 경험이 있어서 시간 관리에는 자신이 있습니다. 디프만 활동에 필요한 시간은 반드시 확보하겠습니다.

**Q. 되고 싶은 개발자 상은?**
> **"기술로 문제를 해결하되, 사용자 경험을 항상 최우선으로 생각하는 개발자"**입니다. 두바이 왕자에서 토스 심사 리젝 후 사용자 인터랙션을 보강해서 통과한 경험처럼, 기술은 결국 사용자에게 가치를 전달하는 수단이라고 생각합니다. 디프만에서 디자이너와 함께 서비스를 만들면서 이 가치관을 더 깊이 실천하고 싶습니다.

**Q. 디프만에서 지원자님의 어떤 점이 도움이 될까요?**
> 두 가지입니다. 첫째, **빠른 실행력**입니다. 4개월간 7개 서비스를 직접 기획부터 배포까지 완료한 경험이 있어서, 짧은 프로젝트 기간 안에 결과물을 만들어내는 데 자신 있습니다. 둘째, **기술적 폭**입니다. React, Next.js, Firebase, Flutter까지 다양한 스택을 경험했기 때문에 팀에서 기술 선택을 논의할 때 여러 관점에서 의견을 제시할 수 있고, 필요하면 새로운 기술도 빠르게 학습해서 적용할 수 있습니다. 또한 회사에서 백엔드(Django) 경험도 있어서 서버 개발자와의 소통에서도 기술적 맥락을 이해하며 협업할 수 있습니다.

**Q. 프로젝트가 성공하는 데 가장 중요한 요소는?**
> **"빠르게 출시하고, 피드백을 받고, 개선하는 사이클"**이라고 생각합니다. 완벽한 기획이나 기술보다 실제 사용자 앞에 내놓는 것이 먼저입니다. 두바이 왕자가 좋은 예인데, 첫 토스 심사에서 리젝됐지만 피드백을 반영해 인터랙션을 보강한 후 통과했습니다. 처음부터 완벽하려 하면 출시 자체가 늦어지고, 실제 사용자의 반응과 내 예상은 항상 다르기 때문에 빠른 출시 → 피드백 → 개선 루프가 가장 중요합니다.

**Q. 프로젝트에서 정보를 공유하지 않으려는 팀원이 있다면?**
> 먼저 **왜 공유를 안 하는지 이유를 파악**합니다. 바빠서 여유가 없는 건지, 자신의 코드에 자신이 없는 건지, 공유 방법을 모르는 건지에 따라 대응이 다릅니다. 저라면 **먼저 제가 적극적으로 공유하는 모습을 보여서** 분위기를 만들겠습니다. 제 작업 내용을 매일 간단히 정리해서 올리고, 배운 점이나 삽질한 것도 공유하면 자연스럽게 서로 공유하는 문화가 형성된다고 생각합니다. 그래도 개선이 안 되면 1:1로 편하게 이야기를 나누겠습니다.

**Q. 디프만에서 내가 발휘할 수 있는 역량은?**
> **"실행력과 기록 문화"**입니다. 혼자 프로젝트를 7개 만들면서 각 프로젝트마다 README에 기술 스택, 구조, 트러블 슈팅을 꼼꼼히 정리해왔습니다. 디프만에서도 프로젝트 진행 과정을 문서화하고, 기술적 의사결정의 이유를 기록하면 팀 전체의 맥락 공유에 기여할 수 있다고 생각합니다. 또한 빠르게 프로토타입을 만들어서 보여주는 것에 강점이 있어서, 초기 기획 단계에서 동작하는 화면을 빠르게 구현해 팀의 방향성을 잡는 데 도움이 될 수 있습니다.

**Q. 나와 맞지 않는 성향은? 그런 사람은 어떻게 대처하나요?**
> **이유 없이 부정하는 성향**이 가장 어렵습니다. "그건 안 될 것 같아요"라고만 하고 대안을 제시하지 않는 경우입니다. 저는 어떤 의견이든 근거가 있으면 존중하지만, 근거 없는 반대는 프로젝트 진행을 막습니다. 이런 경우 **"그러면 어떤 방식이 더 좋을까요?"**라고 질문을 전환해서 대안을 함께 찾는 방향으로 유도합니다. 결국 목표는 좋은 서비스를 만드는 것이기 때문에, 감정이 아니라 **근거 기반으로 논의**하는 것이 중요하다고 생각합니다.

**Q. 당신은 리더인가 팔로워인가?**
> **상황에 따라 유연하게 전환하는 편**입니다. 혼자 7개 프로젝트를 기획부터 배포까지 주도한 경험이 있어서 방향을 잡고 이끄는 것에 익숙하지만, 디프만처럼 다양한 전문가가 모인 팀에서는 각 분야의 전문성을 존중하고 따르는 것도 중요합니다. 기술적 의사결정이 필요할 때는 적극적으로 의견을 내고, 디자인이나 기획 영역에서는 해당 분야 팀원의 판단을 신뢰하며 지원하겠습니다. **팀에 필요한 역할이 무엇이든 채울 수 있는 사람**이 되고 싶습니다.

## 프로젝트 기술 스택 심화 질문

### Vite

**Q. Vite가 뭔가요? CRA(Create React App)와 뭐가 다른가요?**
> 둘 다 React 프로젝트를 만들어주는 **도구**입니다. `npx create-react-app my-app` 또는 `npm create vite@latest my-app`을 실행하면 React 프로젝트의 초기 폴더 구조, 설정 파일, 개발 서버 등을 자동으로 세팅해줍니다.
>
> **CRA (Create React App)**: React 공식 도구였지만 현재는 더 이상 권장되지 않습니다. 내부적으로 **Webpack**이라는 번들러를 사용하는데, 개발 서버를 시작할 때 프로젝트의 **모든 파일을 한꺼번에 번들링(하나로 합침)**한 뒤 실행합니다. 파일이 많아지면 시작이 느려지고, 코드를 수정해도 반영이 느립니다.
>
> **Vite**: 차세대 빌드 도구로, **ESM(브라우저가 import/export를 직접 이해하는 방식)**을 활용합니다. 모든 파일을 미리 합치지 않고 브라우저가 **필요한 파일만 그때그때 가져오기** 때문에 개발 서버 시작이 수백 ms로 매우 빠릅니다. 코드를 수정하면 해당 파일만 교체하는 **HMR(Hot Module Replacement)**으로 거의 즉시 브라우저에 반영됩니다.
>
> 두바이 왕자, Smoke Trace, Kickoff 모두 Vite로 구성했는데, 파일 저장 → 브라우저 반영이 체감될 정도로 빨라서 개발 생산성이 크게 올랐습니다.

**Q. Vite의 HMR이 뭔가요?**
> Hot Module Replacement의 약자로, 코드를 수정하면 전체 페이지를 새로고침하지 않고 변경된 모듈만 교체하는 기능입니다. React의 상태를 유지한 채로 UI가 갱신돼서 개발 중 상태를 다시 만들 필요가 없습니다. Vite는 ESM 기반이라 변경된 파일만 정확히 교체할 수 있어 Webpack보다 HMR이 훨씬 빠릅니다.

### Tailwind CSS

**Q. Tailwind CSS가 뭔가요? 일반 CSS와 뭐가 다른가요?**
> 유틸리티 퍼스트 CSS 프레임워크입니다. 일반 CSS는 `.card-title { font-size: 18px; font-weight: bold; }` 처럼 클래스명을 직접 만들고 스타일을 정의하지만, Tailwind는 `className="text-lg font-bold"`처럼 미리 정의된 유틸리티 클래스를 조합합니다. 클래스명을 고민할 필요가 없고, 사용하지 않는 CSS는 빌드 시 자동으로 제거(purge)돼서 번들 크기가 작습니다.

**Q. Tailwind CSS v4에서 달라진 점은?**
> 두바이 왕자와 포트폴리오에서 Tailwind v4를 사용했습니다. v4는 Rust 기반 엔진(Oxide)으로 빌드 속도가 크게 빨라졌고, `tailwind.config.js` 없이 CSS 파일 안에서 `@theme`으로 설정을 관리합니다. 또한 `@import "tailwindcss"` 한 줄이면 설정이 끝나서 초기 세팅이 훨씬 간결해졌습니다.

**Q. CSS Modules와 Tailwind CSS를 각각 언제 쓰나요?**
> Kickoff에서는 CSS Modules를 사용했고, 다른 프로젝트들은 Tailwind를 사용했습니다. CSS Modules는 컴포넌트마다 스코프가 분리돼서 클래스명 충돌이 없고, 복잡한 커스텀 스타일링에 유리합니다. Kickoff는 경기 스코어보드, 대진표 등 복잡한 레이아웃이 많아서 CSS Modules가 더 적합했습니다. 반면 Tailwind는 빠른 프로토타이핑과 일관된 디자인 시스템 유지에 좋아서 빠르게 만들어야 하는 프로젝트에 사용했습니다.

### Framer Motion

**Q. Framer Motion이 뭔가요? CSS 애니메이션과 뭐가 다른가요?**
> React에서 선언적으로 애니메이션을 만들 수 있는 라이브러리입니다. CSS 애니메이션은 `@keyframes`를 직접 정의해야 하고 JavaScript 상태와 연동이 어렵지만, Framer Motion은 `<motion.div animate={{ opacity: 1 }}>`처럼 React 컴포넌트의 props로 애니메이션을 제어합니다. 포트폴리오에서 스크롤 시 섹션이 fade-in되는 효과, Smoke Trace에서 연기 파티클 효과에 사용했습니다.

**Q. Framer Motion의 whileInView는 어떻게 동작하나요?**
> 요소가 뷰포트에 들어올 때 자동으로 애니메이션을 실행합니다. 내부적으로 IntersectionObserver를 사용합니다. 포트폴리오에서 `<motion.div initial={{ opacity: 0, y: 20 }} whileInView={{ opacity: 1, y: 0 }} viewport={{ once: true }}>`로 스크롤 시 한 번만 나타나는 효과를 구현했습니다. `once: true`가 없으면 스크롤을 왔다갔다 할 때마다 반복됩니다.

### React Router

**Q. Kickoff에서 React Router v7을 어떻게 사용했나요?**
> Kickoff는 홈/순위/일정/리그상세/팀상세/선수상세/경기상세 등 화면이 많은 SPA입니다. React Router v7로 URL 기반 라우팅을 구성하고, `/matches/:id`, `/teams/:id`, `/players/:id` 같은 동적 파라미터로 상세 페이지를 처리했습니다. 뒤로가기와 딥링크도 자연스럽게 동작합니다.

**Q. SPA에서 라우팅이 왜 필요한가요?**
> SPA는 페이지 전체를 새로고침하지 않고 JavaScript로 화면을 전환합니다. 하지만 라우팅 없이 state만으로 화면을 전환하면 URL이 바뀌지 않아서 브라우저 뒤로가기가 안 되고, 특정 페이지를 공유하거나 북마크할 수 없습니다. Kickoff에서 경기 상세 페이지를 친구에게 공유하려면 고유 URL이 필수적이었습니다.

### Recharts

**Q. Kickoff에서 Recharts를 어떻게 사용했나요?**
> 리그 통계 페이지에서 팀별 폼 차트(최근 경기 승패 추이), 경기당 골 수 비교 등을 시각화하는 데 사용했습니다. Recharts는 React 컴포넌트 기반이라 `<LineChart>`, `<BarChart>` 같은 태그로 선언적으로 차트를 그릴 수 있고, 반응형도 지원합니다. D3.js보다 학습 곡선이 낮고 React와 통합이 자연스러워서 선택했습니다.

### Leaflet

**Q. 충주 프로젝트에서 Leaflet을 어떻게 사용했나요?**
> 충주시 12개 행정동을 지도 위에 시각화하기 위해 Leaflet을 사용했습니다. OpenStreetMap 타일 위에 GeoJSON 데이터로 각 행정동 경계를 그리고, 활력도에 따라 색상을 다르게 표시했습니다. 클릭하면 해당 구역의 리워드 정보가 팝업으로 나타나고 체크인할 수 있습니다.

**Q. Google Maps 대신 Leaflet을 선택한 이유는?**
> Google Maps는 API 키 발급과 과금 정책이 있고, 커스터마이징에 제약이 있습니다. Leaflet은 오픈소스이고 완전 무료이며, 행정동 경계를 GeoJSON으로 자유롭게 그릴 수 있어서 선택했습니다. 충주 프로젝트는 12개 행정동별 활력도 색상을 동적으로 바꿔야 했는데, Leaflet이 이런 커스텀 오버레이에 훨씬 유연했습니다.

### Vanilla JS vs Framework

**Q. 충주 프로젝트는 왜 React 없이 Vanilla JS로 만들었나요?**
> 충주 프로젝트는 청년창업 아이디어 캠프 수상 후 데모용으로 빠르게 만든 프로토타입입니다. 페이지가 1개이고 복잡한 상태 관리가 필요 없었기 때문에 프레임워크 없이 HTML/CSS/JS만으로 충분했습니다. LocalStorage로 데이터를 저장하고, DOM 조작도 간단한 수준이라 React를 쓰면 오히려 오버엔지니어링이었을 겁니다.

**Q. 어떤 기준으로 프레임워크 사용 여부를 결정하나요?**
> 페이지 수, 상태 관리 복잡도, 컴포넌트 재사용 필요성을 기준으로 판단합니다. 충주처럼 단일 페이지 데모는 Vanilla JS로 충분하고, Kickoff처럼 여러 페이지에서 공유 상태와 복잡한 데이터 흐름이 있으면 React + Router + TanStack Query 조합이 필요합니다. 포트폴리오는 SEO와 SSG가 필요해서 Next.js를 선택했습니다.

### Firebase 심화

**Q. Firebase Presence 시스템은 어떻게 동작하나요?**
> Firebase Realtime Database의 `.info/connected` 경로를 구독하면 클라이언트의 연결 상태를 감지할 수 있습니다. 연결 시 고유 세션 키를 등록하고, `onDisconnect().remove()`를 설정하면 브라우저를 닫거나 네트워크가 끊겼을 때 서버 측에서 자동으로 해당 키를 삭제합니다. 두바이 왕자와 Smoke Trace의 실시간 동시 접속자 표시에 이 방식을 사용했습니다.

**Q. Firebase Realtime Database에서 데이터 구조를 어떻게 설계했나요?**
> 두바이 왕자의 경우 `/presence/{sessionId}` 에 접속자 정보를, `/stats/today`와 `/stats/total`에 일간/누적 통계를 저장했습니다. Realtime Database는 깊은 중첩 구조보다 평탄한(flat) 구조가 성능에 좋습니다. 데이터를 읽을 때 하위 노드를 전부 가져오기 때문에, 불필요한 데이터 전송을 줄이려면 경로를 분리하는 게 중요합니다.

### 토스 앱인토스(Apps-in-Toss) SDK

**Q. 토스 미니앱 SDK는 어떤 기능을 제공하나요?**
> 토스 앱 안에서 웹앱을 실행할 수 있게 해주는 SDK입니다. `toss.share()`로 토스 내 공유, `toss.close()`로 미니앱 종료, `toss.getUser()`로 토스 사용자 정보 접근 등의 API를 제공합니다. 두바이 왕자와 Kickoff에서 완식/경기 공유 기능에 토스 공유 API를 활용했습니다. 일반 웹에서도 동작하도록 SDK 없을 때는 클립보드 복사로 폴백 처리했습니다.

### Vercel 배포

**Q. Vercel로 배포하는 과정을 설명해주세요.**
> GitHub 레포지토리를 Vercel에 연결하면, main 브랜치에 push할 때마다 자동으로 빌드 + 배포됩니다. PR을 올리면 Preview 배포가 생성돼서 머지 전에 미리 확인할 수 있습니다. 환경 변수는 Vercel 대시보드에서 설정하고, 도메인도 커스텀으로 연결할 수 있습니다. 7개 프로젝트 모두 Vercel로 배포했고, 무료 플랜으로 충분했습니다.

**Q. Vercel과 Netlify의 차이는?**
> 둘 다 정적 사이트 + 서버리스 함수 호스팅이 가능하지만, Vercel은 Next.js를 만든 회사라서 Next.js 프로젝트의 SSR/SSG/ISR을 네이티브로 지원합니다. Edge Functions, 이미지 최적화 등 Next.js 전용 기능이 Vercel에서 가장 잘 동작합니다. 포트폴리오가 Next.js 15 기반이라 Vercel이 최적이었습니다.

### Next.js 심화

**Q. Next.js 15의 App Router와 Pages Router의 차이는?**
> Pages Router는 `pages/` 디렉토리 기반으로 `getServerSideProps`, `getStaticProps`로 데이터를 가져옵니다. App Router는 `app/` 디렉토리 기반으로 React Server Components를 사용하고, `fetch()`에 캐싱 옵션을 직접 설정합니다. 포트폴리오는 App Router를 사용했고, 레이아웃 중첩, 로딩 UI, 에러 바운더리 등을 파일 기반으로 간결하게 처리할 수 있었습니다.

**Q. 포트폴리오에서 API Route를 어떻게 활용했나요?**
> `src/app/api/screenshots/route.ts`에서 프로젝트별 스크린샷 파일을 자동으로 탐색하는 API를 만들었습니다. `public/images/projects/screenshots/{프로젝트명}/` 폴더에 이미지를 넣기만 하면 API가 파일 목록을 반환하고, 모달에서 자동으로 갤러리를 생성합니다. 이미지를 추가할 때 코드 수정이 필요 없는 구조입니다.

**Q. Next.js란 무엇인가요? React와 뭐가 다른가요?**
> React는 UI를 만드는 **라이브러리**이고, Next.js는 React 위에 얹어진 **프레임워크**입니다. React만으로는 라우팅, SSR, 빌드 설정 등을 직접 구성해야 하지만, Next.js는 이것들을 **기본 내장**으로 제공합니다. 비유하면 React가 엔진이라면, Next.js는 엔진 + 차체 + 핸들이 조립된 완성차입니다.

**Q. Next.js의 파일 기반 라우팅이란?**
> `app/` 폴더 안의 **폴더 구조가 곧 URL**이 되는 방식입니다. React Router처럼 코드로 라우팅을 설정할 필요가 없습니다. 예를 들어 `app/about/page.tsx`를 만들면 자동으로 `/about` URL이 생성됩니다. 포트폴리오에서 `app/page.tsx`가 메인 페이지, `app/api/screenshots/route.ts`가 API 엔드포인트로 자동 매핑됩니다.

**Q. 하이드레이션(Hydration)이란?**
> 서버에서 만든 **정적 HTML에 React를 연결하는 과정**입니다. 순서: 1) 서버가 완성된 HTML 전송 → 사용자가 바로 화면을 봄 2) JavaScript 로드 완료 → React가 기존 HTML에 이벤트 리스너, 상태 등을 연결(하이드레이션) 3) 이후 SPA처럼 동작. 즉 **SSR의 빠른 초기 로딩 + SPA의 부드러운 인터랙션** 두 가지를 모두 가져가는 핵심 메커니즘입니다.

**Q. ISR(Incremental Static Regeneration)이란?**
> SSG와 SSR의 **중간 방식**입니다. 빌드 시 정적 HTML을 생성하되(SSG처럼), 설정한 시간이 지나면 **백그라운드에서 페이지를 다시 생성**합니다. 예를 들어 `revalidate: 60`으로 설정하면, 60초마다 최신 데이터로 HTML이 갱신됩니다. 매 요청마다 서버가 렌더링하는 SSR보다 성능이 좋고, 정적이기만 한 SSG보다 데이터가 신선합니다. 포트폴리오는 콘텐츠가 거의 안 바뀌어서 SSG로 충분했지만, 자주 업데이트되는 블로그 같은 사이트라면 ISR이 적합합니다.

**Q. React Server Components(RSC)란?**
> Next.js 13+ App Router에서 도입된 개념으로, **서버에서만 실행되는 컴포넌트**입니다. 기본적으로 App Router의 모든 컴포넌트는 서버 컴포넌트이며, 클라이언트에서 실행이 필요한 컴포넌트(useState, onClick 등 사용)만 파일 상단에 `"use client"`를 선언합니다. 서버 컴포넌트는 JS 번들에 포함되지 않아 **번들 크기가 줄어들고**, 서버에서 직접 DB나 API에 접근할 수 있습니다.

**Q. "use client"는 왜 필요한가요?**
> App Router에서 기본이 서버 컴포넌트이기 때문에, **브라우저에서 동작해야 하는 코드**(useState, useEffect, onClick 등)를 쓰려면 `"use client"`를 선언해야 합니다. 포트폴리오에서 스크롤 애니메이션(Framer Motion), 모달 열기/닫기(useState), 다크모드 토글 같은 인터랙티브 컴포넌트에 `"use client"`를 사용했고, 데이터만 표시하는 컴포넌트는 서버 컴포넌트로 유지해 번들 크기를 줄였습니다.

**Q. Next.js의 Image 컴포넌트가 일반 img 태그와 다른 점은?**
> Next.js의 `<Image>`는 자동으로 **이미지 최적화**를 해줍니다. 1) 브라우저에 맞는 포맷(WebP, AVIF)으로 자동 변환 2) 지정한 크기로 리사이징 3) **lazy loading** 기본 적용 (뷰포트에 들어올 때 로드) 4) 레이아웃 시프트 방지 (width/height 필수). 일반 `<img>`는 이걸 전부 직접 처리해야 합니다. 포트폴리오에서 프로젝트 썸네일, 스크린샷 등에 Image 컴포넌트를 사용해 자동 최적화를 적용했습니다.

**Q. Next.js에서 SEO를 어떻게 처리하나요?**
> App Router에서는 `metadata` 객체나 `generateMetadata` 함수로 페이지별 메타데이터를 설정합니다. `title`, `description`, `og:image` 등을 설정하면 검색엔진 크롤러와 SNS 공유 시 정보가 표시됩니다. SSG/SSR로 완성된 HTML이 전달되기 때문에 크롤러가 콘텐츠를 읽을 수 있어 **CSR보다 SEO에 유리**합니다. 포트폴리오에 메타데이터를 설정해 구글 검색과 카카오톡 공유 시 제목과 설명이 표시되도록 했습니다.

**Q. Next.js의 layout.tsx와 page.tsx의 역할은?**
> `page.tsx`는 해당 URL의 **실제 페이지 콘텐츠**이고, `layout.tsx`는 여러 페이지에서 **공유되는 레이아웃**(헤더, 푸터, 사이드바 등)입니다. layout은 페이지가 전환되어도 **리렌더링되지 않고 유지**됩니다. 포트폴리오에서 `app/layout.tsx`에 폰트 설정, 메타데이터, 공통 스타일을 정의하고, `app/page.tsx`에 메인 콘텐츠를 작성했습니다. 중첩 레이아웃도 가능해서 섹션별로 다른 레이아웃을 적용할 수 있습니다.

### Flutter / 모바일 (FootHub 관련)

**Q. Flutter와 React Native의 차이는?**
> Flutter는 Dart 언어를 사용하고 자체 렌더링 엔진(Skia)으로 UI를 그려서 네이티브 컴포넌트에 의존하지 않습니다. React Native는 JavaScript/TypeScript를 사용하고 네이티브 컴포넌트를 브릿지로 연결합니다. Flutter가 UI 일관성과 성능에서 유리하고, React Native는 웹 개발 경험을 활용하기 좋습니다. FootHub는 커스텀 UI(피치뷰, 스코어보드)가 많아서 Flutter를 선택했습니다.

**Q. FootHub에서 Riverpod과 GoRouter를 왜 선택했나요?**
> Riverpod은 Flutter의 전역 상태관리 라이브러리로, 위젯 트리 밖에서 독립적으로 동작해 테스트가 쉽고 같은 타입의 Provider를 구분할 수 있습니다. GoRouter는 선언적 라우팅 라이브러리로, `/matches/:id/lineup` 같은 중첩 URL과 딥링크를 깔끔하게 처리합니다. React에 비유하면 Riverpod은 Context + React Query, GoRouter는 React Router와 유사합니다.

### 공통 / 개발 환경

**Q. ESLint, Prettier 같은 린터를 사용하나요?**
> 네, 모든 React 프로젝트에서 ESLint + Prettier를 사용합니다. ESLint는 코드 품질(미사용 변수, 잘못된 import 등)을, Prettier는 코드 포맷(들여쓰기, 세미콜론 등)을 관리합니다. 혼자 개발해도 일관된 코드 스타일을 유지하는 게 나중에 유지보수할 때 도움이 됩니다.

**Q. 환경 변수는 어떻게 관리하나요?**
> `.env` 파일에 Firebase API 키, 외부 API 키 등을 저장하고, `.gitignore`에 추가해서 GitHub에 올라가지 않게 합니다. Vite에서는 `VITE_` 접두사, Next.js에서는 `NEXT_PUBLIC_` 접두사가 붙어야 클라이언트에서 접근 가능합니다. 배포 시에는 Vercel 대시보드에서 환경 변수를 별도로 설정합니다.
