## 디프만 지원서

### Q1. 디프만에 지원하게 된 동기와 이번 기수 활동을 통해 이루고 싶은 구체적인 목표를 작성해 주세요. (500자)

산업기능요원으로 서울소프트에서 Django 기반 ERP 시스템을 풀스택으로 개발하며 실무 역량을 쌓고 있습니다. 동시에 개인적으로 React, TypeScript, Next.js를 활용해 4개월간 7개의 웹 서비스를 기획부터 배포까지 완성했습니다. 토스 앱인토스 미니앱 배포, Firebase 실시간 시스템 구축 등 빠르게 만들고 배포하는 경험은 충분히 쌓았지만, 대부분 1인 개발이었기에 체계적인 팀 협업 경험이 부족하다는 한계를 느꼈습니다. 해커톤에서 2인 팀으로 장려상을 수상하며 협업의 가치를 체감했고, 디자이너·개발자가 함께 서비스를 릴리즈하는 디프만이야말로 이 갈증을 해소할 수 있는 환경이라 확신하여 지원하게 되었습니다.
이번 기수를 통해 이루고 싶은 목표는 다양한 직군의 팀원들과 소통하며 사용자 관점의 서비스 설계 역량을 키우는 것입니다. 저는 웹 개발 전반의 실무 경험과 빠른 실행력을 팀에 기여하고, 팀원들로부터 새로운 시각과 자극을 받으며 함께 성장하고 싶습니다.

### Q2. 디프만은 약 4개월간 몰입도 높은 팀 프로젝트를 진행합니다. 시간 관리 계획을 작성해 주세요. (500자)

현재 병행 중인 활동은 (주)서울소프트 근무(산업기능요원)이며, 건국대학교 서울캠퍼스는 휴학 중입니다. 별도 스터디나 사이드 프로젝트 계획은 없어 디프만 활동에 온전히 집중할 수 있습니다.
회사는 평일 자율출근제로 운영되어 출퇴근 시간을 유연하게 조절할 수 있고, 필요 시 연차를 활용해 일정을 조율할 수 있습니다. 지난 기간동안 회사 업무와 개인 프로젝트 7개를 병행하며 배포한 경험이 있기에, 시간 관리에 자신이 있습니다. 당시에도 평일 업무 후 저녁과 주말 시간을 활용하는 루틴이 이미 체화되어 있었습니다.
구체적인 계획은 다음과 같습니다. 평일에는 업무를 집중 처리 및 필요 시 동아리 프로젝트에 참여하며, 주말은 정기 세션 참석과 팀 프로젝트에 할애하고자 합니다.
시간이 부족하다는 핑계 대신 새벽과 주말을 활용하는 것이 이미 습관이 된 만큼, 끝까지 책임감 있게 참여하겠습니다.

### Q3. 예상치 못한 어려움에 직면했지만, 끝까지 책임을 다해 완주했던 경험을 작성해 주세요. (500자)

코레일 해커톤에서 2인 팀으로 참가했을 때, 추천 정확도를 높이기 위해 사용자에게 많은 정보를 입력받는 백엔드 구조를 설계했습니다. 그런데 프론트엔드 개발자가 입력 항목이 많아질수록 사용자 경험이 크게 저하된다고 강하게 반대했습니다. 실제로 사진 촬영 전 단계에서 입력해야 할 정보가 과도해지며 서비스 흐름이 끊기는 문제가 발생했습니다.
각자 왜 그 방식이 좋다고 생각하는지 근거를 솔직하게 공유하고, 현실적인 해결책을 함께 모색했습니다. 단순히 UI에서 항목을 줄이는 것이 아니라 데이터 흐름 자체를 재설계하기로 했습니다. 더 다양한 정보를 가지고 있는 API를 도입해 추천 범위를 먼저 축소하고, 기존 공공 데이터 API와 결합하는 구조로 변경했습니다. 그 결과 사용자 입력 부담을 줄이면서도 추천 정확도와 응답 속도를 함께 개선할 수 있었습니다.
이 경험을 통해 의견이 다를 때 감추지 않고 근거와 함께 이야기하는 소통이 팀의 신뢰를 쌓고, 더 좋은 결과물로 이어진다는 것을 배웠습니다.

### Q4. 주어진 역할의 범위를 넘어, 주도적으로 해결했거나 품질을 높였던 경험 (700자)

서울소프트에서 운수사 ERP 시스템을 유지보수하던 중, 라이브 서버에서만 배차 저장과 기사 교체가 실패하는 문제가 발생했습니다. 이 모듈은 제가 담당하는 영역이 아니었지만, 고객사 업무가 마비되는 긴급 상황이었기에 직접 원인 파악에 나섰습니다.
로컬에서는 정상 동작했고 서버 로그에도 500 에러가 남지 않아 원인 추적이 어려웠습니다. 브라우저 네트워크 탭을 하나하나 확인한 끝에, 요청 자체가 애플리케이션 레이어에 도달하기 전 Nginx에서 414 URI Too Long 에러로 차단되고 있음을 발견했습니다. 원인은 HTMX가 방대한 기사 ID 배열을 GET 방식으로 전송하며 URL 길이가 한도를 초과한 것이었습니다.
해결을 위해 세 가지 작업을 진행했습니다. 첫째, 요청 방식을 GET에서 POST로 전환하고 Django 뷰를 수정했습니다. 둘째, JavaScript에서 Set을 활용한 중복 제거 로직을 추가해 전송 데이터 크기를 줄였습니다. 셋째, 쿼리스트링 리다이렉트 대신 HTMX의 HX-Redirect 헤더를 활용해 처리 방식을 개선했습니다.
이 수정 이후 해당 모듈의 장애가 완전히 해소되었습니다. 해당 프로젝트의 담당자가 아니었지만 고객사의 불편을 방치할 수 없었습니다. 이후 팀 내에서 데이터 규모에 따른 환경 차이를 점검하는 코드 리뷰 항목이 추가되는 계기가 되었습니다.

### Q5. 빠른 실행과 시도를 통해 더 나은 결과를 만들었던 경험 (700자)

두바이 쫀득쿠키가 유행하던 시기, '먹방 체험형 웹앱을 만들면 재밌겠다'는 아이디어가 떠올랐습니다. 트렌드는 빠르게 식기 때문에 완벽한 기획서를 작성하는 대신, 당일 바로 React와 TypeScript로 개발에 착수했습니다. 핵심 기능인 터치 인터랙션과 애니메이션만 먼저 구현하고, 이틀 만에 첫 버전을 배포했습니다.
배포 후 토스 앱인토스 심사에서 첫 리젝을 받았습니다. 콘텐츠 완성도와 사용자 경험이 부족하다는 피드백이었습니다. 이를 계기로 Firebase Realtime Database를 연동해 실시간 동시 접속자와 통계 시스템을 추가했고, 파티클 애니메이션과 동적 대사 시스템으로 인터랙션을 개선했습니다. 수정 후 재심사를 통과해 일주일 만에 정식 배포할 수 있었습니다.
이 경험이 발판이 되어 '빠르게 만들고, 피드백을 반영해 개선하는' 사이클을 반복했습니다. 이후 Smoke Trace, Kickoff 등 여러 서비스를 같은 방식으로 완성하며, 매번 이전 프로젝트의 시스템을 확장·응용하고 새로운 기술적 도전을 더했습니다. Firebase 실시간 시스템을 Smoke Trace에서 확장했고, FootHub의 API 연동 경험을 Kickoff에서 더 복잡한 구조로 발전시켰습니다.
이 과정에서 완벽한 계획은 존재하지 않는다는 것을 배웠습니다. 빠르게 실행하고 피드백으로 방향을 수정하는 것이 더 좋은 서비스를 만든다는 것을 체득했습니다.

### Q6. 웹 개발 과정에서 중요하다고 생각하는 기술적 요소 하나를 선택해 주세요. (최대 1000자)

웹 개발에서 가장 중요하다고 생각하는 요소는 '상태 관리와 데이터 패칭의 설계'입니다. UI가 아무리 좋아도 데이터 흐름이 꼬이면 사용자 경험은 무너지기 때문입니다.
Kickoff 프로젝트에서 이를 가장 깊이 고민했습니다. 유럽 5대 리그, K리그, UEFA 대회 등 800개 이상의 리그 데이터를 외부 API에서 가져와 실시간 스코어, 순위표, 선수 통계, 대진표 등 다양한 화면에 뿌려야 했습니다. 단순히 각 컴포넌트에서 fetch를 호출하면 동일한 데이터를 중복 요청하게 되고, 페이지 전환 시마다 로딩이 반복되어 사용성이 크게 떨어졌습니다.
이를 해결하기 위해 TanStack React Query를 도입했습니다. staleTime과 cacheTime을 데이터 특성에 맞게 분리 설정했습니다. 실시간 경기 스코어는 30초마다 자동 리페칭하고, 리그 순위나 선수 프로필처럼 변동이 적은 데이터는 캐시를 길게 유지해 불필요한 API 호출을 줄였습니다. 또한 queryKey를 리그·날짜·팀 단위로 세분화하여 한 화면에서 데이터를 갱신해도 다른 화면의 캐시가 영향받지 않도록 설계했습니다.
두바이 왕자와 Smoke Trace에서는 다른 관점의 상태 관리를 경험했습니다. Firebase Realtime Database로 동시 접속자 수와 누적 통계를 실시간 동기화해야 했는데, 여러 사용자가 동시에 값을 갱신할 때 데이터 정합성 문제가 발생했습니다. Firebase Transaction을 적용해 원자적 업데이트를 보장했고, onDisconnect 핸들러로 브라우저 종료 시에도 접속자 수가 정확히 차감되도록 구현했습니다.
이 경험들을 통해 상태 관리는 단순히 라이브러리 선택이 아니라, 데이터의 성격과 갱신 주기, 공유 범위를 분석하고 적절한 전략을 설계하는 것이 핵심이라는 것을 배웠습니다. 서버 상태와 클라이언트 상태를 명확히 분리하고, 각각에 맞는 도구와 패턴을 적용하는 것이 안정적이고 반응성 높은 웹 서비스의 기반이라고 생각합니다.

### Q7. 여러 번의 시행착오 끝에 구현에 성공한 기능이 있다면 소개해 주세요. (최대 1000자)

두바이 왕자 프로젝트에서 실시간 동시 접속자 수를 표시하는 기능을 구현하며 가장 많은 시행착오를 겪었습니다.
첫 번째 시도는 단순하게 접근했습니다. 사용자가 접속하면 Firebase에 카운트를 +1하고, 페이지를 떠나면 -1하는 방식이었습니다. 그러나 브라우저를 그냥 닫거나 네트워크가 끊기면 -1 이벤트가 발생하지 않아 접속자 수가 계속 누적되는 문제가 생겼습니다. 테스트에서는 동시 접속자가 50명으로 표시되는데 실제로는 아무도 없는 상황이 발생했습니다.
두 번째 시도에서 Firebase의 Presence 시스템을 도입했습니다. onDisconnect 핸들러를 등록하면 서버 측에서 연결 해제를 감지해 자동으로 정리해주는 구조입니다. 그런데 이번에는 동시 접속자 수가 음수로 떨어지는 새로운 문제가 나타났습니다. 여러 탭을 열었다가 동시에 닫으면 onDisconnect가 중복 실행되며 카운트가 과도하게 차감된 것이었습니다.
세 번째 시도에서 근본적으로 구조를 바꿨습니다. 숫자를 단순 증감하는 대신, 각 사용자 세션을 고유 키로 등록하고 연결이 끊기면 해당 키를 삭제하는 방식으로 전환했습니다. 동시 접속자 수는 등록된 키의 개수로 계산합니다. 여기에 Firebase Transaction을 적용해 여러 사용자가 동시에 통계를 갱신할 때도 데이터 정합성이 깨지지 않도록 원자적 업데이트를 보장했습니다.
이렇게 완성한 실시간 시스템은 이후 Smoke Trace에서 그대로 확장 적용할 수 있었습니다. 흡연 시뮬레이션의 동시 접속자와 누적 통계에 동일한 패턴을 재사용하되, 개인별 통계 추적 기능을 추가하는 방식으로 발전시켰습니다.
이 경험을 통해 얻은 문제 해결 방식은 '구조를 의심하라'는 것입니다. 첫 번째와 두 번째 시도는 기존 구조 위에서 버그를 패치하려 했지만, 근본적인 데이터 모델 자체를 바꿨을 때 문제가 깔끔하게 해결되었습니다. 코드를 고치기 전에 설계가 올바른지 먼저 점검하는 습관이 이때 생겼습니다.

### Q. 간단히 자기소개

(TAVE, 잇타와 동일)
안녕하십니까, 저는 건국대학교 서울캠퍼스 스마트ICT융합공학과 23학번으로 2학년까지 수료 후, 현재 병역특례 산업기능요원으로 서울소프트라는 회사에서 풀스택 개발자로 근무하고 있습니다. 버스운수사 ERP 시스템 개발, 유지보수를 담당하며 실무 경험을 쌓고 있습니다.
저는 개인 프로젝트, 해커톤, 앱 출시 등 다양한 경험을 해왔습니다. 그러나 그 과정에서 가장 어려웠던 것은 개발보다 디자인, 마켓팅 이었습니다. 아무리 좋은 서비스를 만들어도 이를 예쁘게 포장하고, 사람들에게 알릴 수 없다면 좋은 서비스로 확장해나가기 어렵다고 느꼈습니다. 그렇기에 저는 다양한 역할의 팀원들과의 협업 속에서 어떻게 디자인되고, 마케팅되면 지켜보고 이 또한 배워 보고 싶다는 생각이 들었습니다. 1년간 회사를 다니며 정말 많은 것을 배웠습니다. 다양한 역할의 사람들과 프로젝트를 진행한다면 저 역시 빠르게 성장할거라고 생각하였습니다. (더 있는데 기억안남, 스터디 얘기 한듯)

### Q. 프로젝트 하다가 팀원에게 문제가 생긴다면?

방금 말씀드린 것처럼 이 또한 솔직한 대화가 중요하다고 생각합니다. 만약 본인 역량문제라면 남은 업무에서 할 수 있는 테스크로 쪼개서 제안할 것이고, 의지 문제라면 다른 팀원들이 작업한 것을 볼 수 있는 현황판을 만들어 자극을 받게 할 것입니다.

### Q. 디프만에 어떻게 기여할 수 있을지

빠른 실행력과 기술적 폭으로 가능. 지난 개인 프로젝트 진행하며 처음 사용하는 언어, 프레임워크도 빠르게 적응 가능.

### Q. 자소서 보니 414 에러가 있음. 근데 이거 서버에서 URL 길이 제한 풀면 되는데 왜 굳이 저렇게?

이것이 실제로 서비스하고 있고, 고객사가 있는 프로젝트이다 보니 서버 설정 건드리기에는 부담이 많았음. 다른 곳에서 문제가 터질 수도 있기에, 서버 설정 건드는 거는 지양하는 편. 그래서 HTMX 방식을 수정함. HTMX는 전체를 로드하는 방식이라 화면 깜빡임이 있거나 그럴 수 있음. 그래서 AJAX를 이용함. 이거는 React로 따지면 해당 컴포넌트만 변경하는 것처럼 해당 부분만 바꾸기에 고객사 입장에서도 매우 편리.

### Q. 상태관리, RDB 관련 이슈 해결 어떻게 함?

TanStack Query 사용함. 실제로 Kickoff에서 사용함. 해당 프로젝트는 축구 프로젝트라 실시간 데이터를 주고받음. 그렇기에 staleTime과 gcTime을 사용함. staleTime은 데이터에 유통기한이 있는 느낌, 해당 시간이 지나면 포커스, 구독이 되어도 시간이 지나면 폐기. gcTime은 해당 시간 안에 포커스가 된다면 다시 사용 가능, 시간 초기화. 해당 시간 안에 포커스 되지 않는다면 가비지 되어버림. 그렇기에 축구 데이터의 경우 포커스가 계속 되어도 데이터는 갱신 되어야하기에 staleTime을 이용. 경기 실시간 데이터는 30초, 리그 순위의 경우 5분으로 설정함.

### Q. 전역상태 관리는 어케?

useContext 이용. 이걸 처음 들었을때 전역변수가 떠오름. 상단에서 프로바이더로 감싸면 하단 컴포넌트에서 재사용 가능, 효율적인 구조 가능.

### Q. 개인 프로젝트가 많다보니, 디프만같이 협업 프로젝트가 좀 느릴수 있다. 괜찮냐?

저 혼자 진행하다 보니, 아쉬움이 많았음. 실제 내 프로젝트도 다른 팀원들이 많았으면 분명 여러번 피봇되고 그랬을텐데 그 과정이 없어서 아쉬웠음. 그래서 다른 사람들과 소통해나가며 진행해 보고 싶음. 그런걸 하고싶어서 지원한거임.

### Q. 다른 산기요분들 보면 실력도 좋고 하다. 영찬님도 그런거같은데 매니징 역할 시키면 가능?

오히려 너무 좋다. 책임감을 쥐어 준다면 정말 잘할 스타일임. 그렇기에 시켜만 준다면 가능하다.

## 후기

사실 워낙 유명하고 큰 동아리라 서류 붙여준 것만해도 매우 감사했음. 근데 마지막에 매니징 역할 할 수 있냐고 물어본게 살짝 합격시그널 느낌 나서 기분은 좋았음. 디프만의 경우 서류 통과도 자신감 없어서 안넣었는데, 잇타, TAVE 둘다 된거 보고 자신감 붙어서 술먹은 상태로 급하게 대강 써서 제출했는데 잘하면 진짜 갈수도 있을듯 싶었음. 위 동아리 두개와는 다르게 현직자도 많은 동아리라 배울점이 많을 거 같아서 좋음. 이 면접 준비하면서 React 2~3일 빡세게 공부했는데 좀 재밌었음. 아 그리고 회사 이사님이 최대 강점은 '두바이 왕자' (두쫀쿠 먹는거)라고 하셨는데 면접때 이걸 좋게 본거같긴했음. 일단 재밌고, 이용자가 많은게 좋아보인듯.
